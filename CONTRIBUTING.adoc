= NestJS AI Contributor Guidelines

Do you have something you'd like to contribute to *NestJS AI*?
We welcome pull requests, but ask that you carefully read this document first to understand how best to submit them;
what kind of changes are likely to be accepted; and what to expect when your submission is reviewed.

Please refer back to this document as a checklist before issuing any pull request; this will save time for everyone.

== Understand the basics

Not sure what a *pull request* is, or how to submit one? Take a look at GitHub's documentation:
https://docs.github.com/en/pull-requests[Using Pull Requests].

== Search GitHub issues first; create one if necessary

Is there already an issue that addresses your concern? Search the
https://github.com/nestjs-port/nestjs-ai/issues[GitHub issue tracker] to see if you can find something similar.
If not, please create a new issue before submitting a pull request unless the change is truly trivial, e.g. typo fixes,
documentation corrections, or removing compiler/lint warnings.

== Developer Certificate of Origin

All commits must include a `Signed-off-by` trailer at the end of each commit message to indicate that the contributor
agrees to the Developer Certificate of Origin.

For convenience, use:

[source,shell]
----
git commit --signoff
----

== Fork the repository

1. Go to https://github.com/nestjs-port/nestjs-ai
2. Hit the *Fork* button and choose your own GitHub account as the target
3. For more detail see https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/working-with-forks[Fork a repo]

== Set up your local development environment

1. `git clone --recursive git@github.com:<your-github-username>/nestjs-ai.git`
2. `cd nestjs-ai`
3. `git remote show`
   _you should see only `origin` - which is the fork you created for your own GitHub account_
4. `git remote add upstream git@github.com:nestjs-port/nestjs-ai.git`
5. `git remote show`
   _you should now see `upstream` in addition to `origin`_
6. `git fetch --all`
7. `git branch -a`
   _you should see branches on origin as well as upstream, including `main`_

== A day in the life of a contributor

* Always work on topic branches. Use the GitHub issue ID as the branch name when possible.
* For example, to create and switch to a new branch for issue GH-123: `git checkout -b GH-123`
* You might be working on several different topic branches at any given time, but when you reach a stopping point for
  one of those branches, commit locally.
* If you want to collaborate with another contributor, have them fork your repository (add it as a remote) and
  `git fetch <your-username>` to grab your branch.
* Alternatively, they can use `git fetch --all` to sync their local state with all of their remotes.
* When ready for review, issue a pull request to `nestjs-port/nestjs-ai`.
* A prerequisite for any pull request is that it can be cleanly merged with the current `main` branch.
  This is the responsibility of every contributor.
* If your pull request cannot be applied cleanly, you will most likely be asked to rebase or otherwise make it mergeable.

== Keep your local code in sync

* Always work on topic branches, since `main` is a moving target.
* Keep your local `origin/main` in sync with `upstream/main`.
* Within your local working directory, you can sync up all remotes' branches with: `git fetch --all`
* While on your own local `main` branch: `git pull upstream main`
* Now that you're in sync, switch to the topic branch where you plan to work, e.g. `git checkout -b GH-123`
* If changes have occurred on `upstream/main` while you were working:
  - Switch back to `main`: `git checkout main`
  - Then: `git pull upstream main`
  - Switch back to the topic branch: `git checkout GH-123`
  - Rebase the topic branch to minimize the distance between it and your recently synced `main` branch:
    `git rebase main`
* Once you have published your branch, merge `main` rather than rebasing history.

== Maintain a linear commit history

When merging to `main`, this project uses fast-forward merges where possible.
When issuing pull requests, please ensure that your commit history is linear.

You can check this from the command line with:

[source,shell]
----
git log --graph --pretty=oneline
----

If you see intersecting lines, that usually means you forgot to rebase your branch.

== Run formatting checks and make sure the build passes

Before opening a pull request, make sure the full project build passes locally.

[source,shell]
----
pnpm format
pnpm lint
pnpm test
pnpm build
pnpm docs:build
----

== Source code style

NestJS AI source code follows the project's TypeScript, NestJS, and Biome conventions.
Please preserve existing formatting and naming patterns.

== Mind the whitespace

Please carefully follow the whitespace and formatting conventions already present in the repository.

1. Tabs, not spaces, where the file already uses tabs
2. Unix (LF), not DOS (CRLF) line endings
3. Eliminate all trailing whitespace
4. Preserve existing formatting; do not reformat code for its own sake
5. Search the codebase using `rg` and other tools to discover common naming conventions

== Submit tests for behavior changes

Add or update tests for any behavior change.
When possible, keep test names descriptive and colocate them with the code they exercise.

== Squash commits

Use `git rebase --interactive`, `git add --patch`, and similar tools to squash multiple commits into atomic changes
before opening a pull request.

== Use your real name in git commits

Please configure git to use your real first and last name for any commits you intend to submit as pull requests.

You can configure this globally with:

[source,shell]
----
git config --global user.name "First Last"
git config --global user.email user@mail.com
----

Or locally for the *nestjs-ai* repository only by omitting the `--global` flag.

== Run all tests prior to submission

Make sure that all tests pass prior to submitting your pull request.
CI will run the equivalent of the commands listed above, so make sure they pass locally before opening your PR.

== Mention your pull request on the associated GitHub issue

If your change was tracked by an issue, add a comment to that issue linking to the pull request.

== Provide a link to the GitHub issue in the pull request

Link the pull request to the associated GitHub issue using GitHub's built-in issue linking.
